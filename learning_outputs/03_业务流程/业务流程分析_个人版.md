# 业务流程分析：个人版

填写日期：2026-06-12

学习者：ASP.NET Core 程序员转 AI Solution Consultant

流程名称：客服工单分类与回复建议流程

---

## 1. 业务背景

企业客服每天处理大量客户问题。传统流程中，客服人员需要人工阅读工单、判断类别和优先级、查询知识库、撰写回复并决定是否升级。这个流程高频、重复、标准容易不一致，适合作为 Agentic AI 入门项目。

本流程用于训练：

- 业务流程诊断
- Agent 介入点识别
- 结构化输出
- RAG 检索
- 工具调用
- 人工确认和风险控制

---

## 2. 参与角色

| 角色 | 职责 |
|---|---|
| 客户 | 提交问题或投诉 |
| 一线客服 | 处理普通工单、确认回复内容 |
| 二线客服 | 处理复杂或高风险问题 |
| 客服主管 | 维护分类规则和升级规则 |
| AI Agent | 分类、判断优先级、检索知识库、生成回复建议 |
| IT / 系统管理员 | 维护接口、权限、日志和数据 |

---

## 3. 当前流程 As-Is

```text
客户提交工单
  -> 客服系统记录问题
  -> 一线客服人工阅读工单
  -> 判断问题类别和优先级
  -> 查询知识库或历史工单
  -> 手写回复内容
  -> 判断是否需要升级二线
  -> 发送回复或转交二线
  -> 关闭工单并记录结果
```

---

## 4. 当前痛点

| 痛点 | 影响 | 严重程度 |
|---|---|---|
| 人工分类耗时 | 工单高峰期响应慢 | 中 |
| 分类标准不一致 | 不同客服处理结果不稳定 | 中 |
| 知识库查询分散 | 客服需要反复查资料 | 中 |
| 高风险工单可能遗漏 | 投诉、付款、退款问题处理不及时 | 高 |
| 回复质量不稳定 | 客户体验不一致 | 中 |

---

## 5. AI 可介入节点

| 节点 | AI 可以做什么 | 风险 |
|---|---|---|
| 工单创建后 | 自动识别类别、优先级、是否需要人工 | 误判高风险问题 |
| 查询资料时 | 检索知识库和历史相似工单 | 引用错误或资料过期 |
| 回复前 | 生成回复草稿 | 可能做出不当承诺 |
| 升级判断 | 判断是否需要二线或主管介入 | 漏掉投诉或敏感问题 |
| 处理完成后 | 生成工单摘要和标签 | 摘要遗漏关键信息 |

---

## 6. 人工审批节点

| 节点 | 为什么必须人工确认 |
|---|---|
| 涉及付款、退款、投诉 | 影响客户权益，不能让 Agent 自动给最终结论 |
| 对外正式回复 | AI 只能生成草稿，发送前应由客服确认 |
| 升级或关闭高风险工单 | 防止误关闭或处理不充分 |
| 修改客户资料 | 涉及业务系统写操作，需要权限和日志 |

---

## 7. 改进后流程 To-Be

```text
客户提交工单
  -> 客服系统记录问题
  -> Agent 读取工单内容
  -> Agent 分类、判断优先级和风险等级
  -> Agent 检索知识库或历史工单
  -> Agent 生成回复建议和处理理由
  -> 如果是高风险问题，转人工或二线
  -> 如果是普通问题，客服确认回复
  -> 系统记录 Agent 判断、工具调用、人工确认和最终结果
```

---

## 8. 成功指标

- 工单初步分类准确率达到 80% 以上。
- 普通工单平均初步处理时间降低 30%。
- 高风险工单转人工命中率达到 95% 以上。
- Agent 输出 JSON 格式合规率达到 95% 以上。
- 每个工单都能记录分类理由、工具调用和人工确认状态。
