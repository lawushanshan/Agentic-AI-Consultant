# 工具设计卡片：个人版

填写日期：2026-06-12

学习者：ASP.NET Core 程序员转 AI Solution Consultant

Agent 场景：客服工单分类与回复建议 Agent

---

## 1. 工具设计原则

工具不是把后端 API 原样暴露给 Agent，而是要按业务任务重新封装。

每个工具都必须明确：

- 业务用途
- 允许使用的场景
- 不允许使用的场景
- 输入参数
- 输出结构
- 权限要求
- 风险等级
- 是否需要人工确认
- 失败时怎么处理
- 需要记录哪些日志

---

## 工具 1：查询客户信息

### 工具名称

`get_customer_profile`

### 业务用途

用于客服处理工单时查询客户等级、历史风险标签和联系方式脱敏信息。

### 什么时候允许使用

- 用户已经登录客服系统。
- 当前客服有权限处理该客户工单。
- 仅用于当前工单处理。

### 什么时候不允许使用

- 客服无权访问该客户。
- 用户只是随意查询客户资料。
- Agent 试图批量查询客户信息。

### 输入参数

| 参数 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| customer_id | string | 是 | 客户 ID |
| ticket_id | string | 是 | 当前工单 ID |

### 输出结果

```json
{
  "customer_id": "C001",
  "customer_level": "vip",
  "risk_tags": ["payment_sensitive"],
  "masked_phone": "138****1234"
}
```

### 权限要求

客服本人必须拥有当前工单处理权限。

### 风险等级

中

### 是否需要人工确认

否，但必须记录日志。

### 失败时怎么办

返回权限不足或客户不存在，不允许 Agent 编造客户信息。

### 日志记录

需要记录：

- user_id
- ticket_id
- customer_id
- timestamp
- permission_result

## 工具 2：检索知识库

### 工具名称

`search_support_knowledge_base`

### 业务用途

根据工单内容检索客服 FAQ、售后规则和产品说明。

### 什么时候允许使用

- 处理客服工单时。
- 生成回复建议前。

### 什么时候不允许使用

- 查询超出客服权限的内部资料。
- 查询客户隐私数据。

### 输入参数

| 参数 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| query | string | 是 | 检索问题 |
| user_role | string | 是 | 用户角色 |
| top_k | integer | 否 | 返回片段数量 |

### 输出结果

```json
{
  "results": [
    {
      "chunk_id": "faq_001_003",
      "title": "客服常见问题",
      "content": "付款异常需转人工核查...",
      "score": 0.86
    }
  ]
}
```

### 权限要求

按 user_role 过滤可访问文档。

### 风险等级

低到中

### 是否需要人工确认

否

### 失败时怎么办

提示未检索到可靠资料，建议转人工或联系知识库负责人。

### 日志记录

需要记录：

- user_id
- query
- returned_chunk_ids
- timestamp

## 工具 3：查询历史工单

### 工具名称

`search_similar_tickets`

### 业务用途

查询相似历史工单，辅助判断分类、优先级和处理建议。

### 输入参数

| 参数 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| ticket_text | string | 是 | 当前工单内容 |
| customer_id | string | 否 | 客户 ID |
| limit | integer | 否 | 返回数量 |

### 输出结果

```json
{
  "similar_tickets": [
    {
      "ticket_id": "T1001",
      "category": "payment_issue",
      "resolution_summary": "转人工核查支付流水"
    }
  ]
}
```

### 权限要求

只能返回当前客服有权查看的历史工单，并对敏感信息脱敏。

### 风险等级

中

### 是否需要人工确认

否

### 失败时怎么办

返回空列表，不影响主流程。

### 日志记录

需要记录：

- user_id
- ticket_id
- query_hash
- returned_ticket_ids

## 工具 4：生成回复草稿

### 工具名称

`draft_customer_reply`

### 业务用途

根据分类结果、知识库引用和风险等级生成客服回复草稿。

### 输入参数

| 参数 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| ticket_text | string | 是 | 客户问题 |
| category | string | 是 | 工单类别 |
| knowledge_refs | array | 否 | 引用资料 |
| risk_level | string | 是 | 风险等级 |

### 输出结果

```json
{
  "subject": "关于订单付款状态的处理说明",
  "body": "您好，我们已收到您的反馈...",
  "approval_required": true,
  "reason": "涉及付款异常，需要客服确认后发送"
}
```

### 权限要求

仅生成草稿，不允许直接发送。

### 风险等级

中到高

### 是否需要人工确认

是

### 失败时怎么办

提示无法生成草稿，并转人工处理。

### 日志记录

需要记录：

- user_id
- ticket_id
- draft_hash
- approval_required
- timestamp

## 工具 5：创建升级工单

### 工具名称

`create_escalation_ticket`

### 业务用途

当工单涉及付款、退款、投诉、VIP 客户或复杂问题时，创建二线升级工单。

### 输入参数

| 参数 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| ticket_id | string | 是 | 原工单 ID |
| reason | string | 是 | 升级原因 |
| priority | string | 是 | 优先级 |

### 输出结果

```json
{
  "escalation_id": "E1001",
  "status": "created",
  "assigned_group": "second_level_support"
}
```

### 权限要求

客服或主管角色可以创建升级工单。

### 风险等级

中

### 是否需要人工确认

普通升级不需要；涉及重大投诉或赔付建议需要主管确认。

### 失败时怎么办

保留当前工单状态，提示人工手动升级。

### 日志记录

需要记录：

- user_id
- ticket_id
- escalation_id
- reason
- timestamp

## 工具清单总览

| 工具 | 用途 | 风险等级 | 是否审批 |
|---|---|---|---|
| get_customer_profile | 查询客户信息 | 中 | 否 |
| search_support_knowledge_base | 检索知识库 | 低到中 | 否 |
| search_similar_tickets | 查询相似历史工单 | 中 | 否 |
| draft_customer_reply | 生成回复草稿 | 中到高 | 是 |
