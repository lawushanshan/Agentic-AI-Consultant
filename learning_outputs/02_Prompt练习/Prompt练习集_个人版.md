# Prompt 练习集：个人版

填写日期：2026-06-12

学习者：ASP.NET Core 程序员转 AI Solution Consultant

目标：完成至少 20 个业务 Prompt 案例，并记录改进前后对比。Prompt 练习要服务后续 RAG、工具调用和 Agent 项目，不做纯娱乐类练习。

---

## 1. 练习范围

本阶段重点覆盖：

- 客服工单分类
- 企业知识库问答
- 销售跟进邮件
- 报销或采购初审
- 会议纪要行动项
- 合同条款风险摘要
- 结构化 JSON 输出
- 人工审批判断

---

## 2. Prompt 写法原则

每个 Prompt 尽量包含：

- 角色：让模型知道自己扮演什么业务角色
- 背景：说明业务场景
- 任务：明确要完成什么
- 输入：提供待处理内容
- 输出格式：优先使用 JSON、表格或固定字段
- 限制条件：说明不能编造、不能越权、何时转人工
- 示例：必要时给一个输出样例

### 案例 1：客服工单分类

#### 任务类型

分类 / 风险判断 / 结构化输出

#### 原始 Prompt

```text
请判断下面客户问题的类别和优先级：
客户说：“我已经付款了，但是订单还是显示未支付。”
```

#### AI 输出结果

```text
类别：支付问题 / 订单异常
优先级：高（涉及资金与交易状态不一致，客户存在焦虑情绪，需尽快核实处理，避免影响后续发货或客户信任）
```

#### 存在问题

- 输出格式不稳定。
- 没有说明是否需要人工介入。
- 没有给出业务判断依据。

#### 改进后的 Prompt

```text
你是一名客服主管。
背景是：我们需要把客户工单分配给不同客服小组，并识别高风险问题。
你的任务是：判断下面客户问题的类别、优先级、是否需要人工处理，并给出理由。

客户问题：
“我已经付款了，但是订单还是显示未支付。”

请严格按照 JSON 输出：
{
  "category": "payment_issue | logistics_issue | refund_issue | account_issue | other",
  "priority": "low | medium | high",
  "need_human": true,
  "reason": "判断理由"
}

限制条件：
- 涉及付款、退款、投诉的问题，优先级至少为 medium。
- 涉及付款异常的问题必须 need_human = true。
- 不要编造订单状态，只能根据输入判断。
```

#### 改进后的输出结果

```text
{
"category": "payment_issue",
"priority": "high",
"need_human": true,
"reason": "客户明确反馈已付款但订单状态未更新，属于支付异常问题，涉及资金安全与订单履约，必须由人工客服介入核实支付渠道、对账状态并协调处理，避免客户损失和信任风险。"
}
```

#### 我学到了什么

- 通过角色设定和背景说明能让大模型更好的理解业务问题
- 指定输出条件，能够固化大模型的输出格式
- 通过限制条件 能够对特殊规则或者是规定进行说明，让大模型能够更好的遵守特定的业务逻辑或者是业务规则

### 案例 2：企业制度问答拒答

#### 任务类型

RAG 问答 / 无答案拒答

#### 原始 Prompt

```text
请回答员工年假可以累计几年。
```

#### AI 输出结果

```text
关于员工年假可以累计几年，需要先明确一个核心概念：法律规定的“累计”指的是工作年限的累计，而不是指未休完的年假可以无限期地跨年累积。

具体来说：

1. “累计”指的是工作年限
根据规定，你每年能享受的年假天数，是根据你的累计工作时间来确定的。这个“累计”包括了你在中国境内所有单位的工作时间，换工作并不会导致工龄重新计算。

2. 年假天数按累计工龄计算
累计工作满1年不满10年：每年可休 5天 年假。

累计工作满10年不满20年：每年可休 10天 年假。

累计工作满20年及以上：每年可休 15天 年假。

3. 未休完的年假可以“跨1年”
关于未休完的年假能否累积到明年，规定是：年休假一般不跨年度安排。但单位因生产、工作特点确有必要时，可以跨1个年度安排。

这意味着，今年的年假最多只能顺延到明年，而不能无限期地累计好几年。

4. 如果公司不安排休假
如果公司因工作需要无法安排你休假，应当在本年度内，按你日工资收入的300% 支付未休年休假工资报酬。

如果公司既不安排休假，也不支付报酬，你可以向劳动行政部门投诉，要求其责令公司改正。

总结一下：

工龄可以终身累计，并据此决定你每年的年假天数。

未休的年假一般只能跨1年，不能多年累积。

如果你想了解更具体的情况（例如入职新公司当年的年假如何计算），可以随时再问我。
```

#### 存在问题

- 没有限定必须基于制度文档回答。
- 没有要求引用来源。
- 可能根据常识编造。

#### 改进后的 Prompt

```text

角色：你是企业制度问答助手。
背景：你只能基于提供的制度片段回答问题。如果制度片段中没有答案，请回答“根据当前资料无法确认”，不要编造。

制度片段：
【xxxx公司制度 
一、年假制度
累计工作满1年不满10年：每年可休 5天 年假。
累计工作满10年不满20年：每年可休 10天 年假。
累计工作满20年及以上：每年可休 15天 年假

二、调休制度
日常加班不调休，周末或者是节假日加班才给调休

三、加班餐制度
日常加班有15元餐补，周末加班是60元餐补
】

限制：资料不足的时候，拒绝回答问题

用户问题：
员工年假可以累计几年？

输出格式如下：
答案：
引用来源：
是否能确认：是/否
```

#### 改进后的输出结果

```text
答案：员工年假天数根据累计工作年限确定：累计工作满1年不满10年为5天，满10年不满20年为10天，满20年及以上为15天。

引用来源：制度片段“一、年假制度”中规定：“累计工作满1年不满10年：每年可休 5天 年假。累计工作满10年不满20年：每年可休 10天 年假。累计工作满20年及以上：每年可休 15天 年假。”

是否能确认：是
```

#### 我学到了什么

- 通过明确的输出格式和引用来源，可以有效提高回答的准确性和可靠性。
- 可以让大模型使用外部知识引用，这样可以有效使用企业内部的专业知识或者是领域知识，包裹显性知识和隐形知识

### 案例 3：销售跟进邮件草稿

#### 任务类型

邮件生成 / 人工审批

#### 原始 Prompt

```text
帮我写一封销售跟进邮件。
```

#### AI 输出结果

```text
场景 A：标准跟进（针对已发资料或电话未接）
适用条件： 上次发送了资料但没有回复，或约好的电话时间对方未接。

邮件主题： 关于 [项目名称/产品] 的跟进 | [你的名字]

邮件正文：

[客户名字] 您好，

见字如面。

因为上一封邮件/电话未能与您直接取得联系，所以特地再跟进一下。不知道您是否已经审阅了之前发送的 [方案/报价/资料]？

考虑到 [客户公司名称] 目前在 [行业痛点，如：降本增效/数字化转型] 方面的需求，我特别整理了一个关于 [同行业客户] 的落地案例，希望能为您团队的决策提供一些参考：

效果： 他们在使用我们的方案后，[核心数据提升，如：效率提升了 X% / 成本降低了 X%]。

适配点： 这个方案特别针对 [贵司可能存在的具体问题] 做了优化。

如果您方便的话，今天下午或明天上午是否有 15 分钟的时间？我想简单跟您同步一下这个案例的重点，不会占用太久。

祝工作顺利！

[你的名字] | [职位]
[公司名称] | [联系电话]
[官网链接]
```

#### 存在问题

- 缺少客户背景。
- 缺少语气要求。
- 没有提醒人工确认。
- 容易生成过度承诺。

#### 改进后的 Prompt

```text
你是一名 B2B 销售助理。
背景是：销售需要跟进一个可能续约的企业客户。
你的任务是：根据客户信息生成一封跟进邮件草稿。

客户信息：
- 公司：xx电子商务
- 最近一次采购：2026年10月1日，采购了1000套Ipad
- 已知问题：后续陆续反馈了一些IPAD的电池问题，需要跟进后续的处理进展。
- 跟进目标：预约一次沟通会议

请输出：
邮件主题：
邮件正文：
需要销售人工确认的内容：
会议时间、会议地点、参会人员

限制条件：
- 不要承诺价格、折扣、交付日期。
- 不要替销售直接发送邮件。
- 邮件语气专业、简洁。
```

#### 改进后的输出结果

```text
邮件主题： 关于iPad电池问题处理进展及后续合作沟通 | xx电子商务

邮件正文：

xx电子商务 采购负责人 [如有具体姓名请填写，否则可写“您好”]：

感谢您一直以来对我们的信任与支持。关于贵司于2026年10月1日采购的1000套iPad，我们注意到后续反馈中提到了部分设备的电池使用问题，对此我们非常重视。

目前，我们已就电池问题与技术支持团队进行了专项梳理，并整理出了相应的处理进展和后续跟进方案。考虑到该批设备涉及贵司日常运营使用，且续约周期临近，我们希望就以下事项与您当面沟通：

电池问题的具体分析结果及已采取的补救/更换措施进展；

剩余设备的使用情况回访；

后续续约服务的保障优化建议。

为此，特向您预约一次沟通会议，时间约30分钟，具体安排可根据您的日程调整。

烦请您确认以下信息，或告知您方便的时段，我会尽快协调落实。

期待与您深入交流，确保后续服务更加顺畅。

顺颂商祺！

[你的姓名]
[你的职位] | [公司名称]
[联系电话] | [邮箱地址]

需要销售人工确认的内容：

项目	待确认信息
会议时间	建议由销售先提供2~3个备选时段（如：7月15日（周三）上午10:00、7月16日（周四）下午2:00、7月17日（周五）上午9:30），请客户选择，或请客户直接告知其方便时间。
会议地点	请确认：线上（腾讯会议/Teams/钉钉，并提前准备链接）或 客户公司现场 / 我方公司会议室。建议默认提供线上+客户现场两种选项供客户选择。
参会人员	请确认我方参会人员（是否需技术支持或售后负责人一同参加），并请客户告知其参会人员名单（如采购负责人、使用部门主管等）。
⚠️ 销售人工操作提醒（请勿发送以下内容给客户）：

请勿在邮件中主动提及具体折扣、续约价格或交货周期，如有涉及需在会议中面谈。

发送前请务必确认客户联系人姓名拼写和公司名称准确。

该邮件仅为预约沟通，不构成任何正式承诺。
```

#### 我学到了什么

- 可以通过明确的角色、背景和任务，让大模型更好地理解业务场景，生成更符合实际需求的邮件内容。
- 通过限制条件和人工确认提醒，可以有效避免大模型生成过度承诺
- 可以通过提示的形式，让大模型明确哪些内容可以直接输出，哪些内容需要人工二次确认，从而提高邮件的专业性和准确性。

### 4.1 实操案例

案例 4：报销申请风险判断

你是一名报销初审助手。仅根据输入材料做初步风险标记，不能批准报销。
请严格输出 JSON：
{
"risk_level": "low | medium | high",
"risk_tags": ["重复报销 | 超标准 | 发票异常 | 材料缺失 | 无明显风险"],
"need_human_approval": true,
"reason": "不超过80字",
"missing_materials": []
}

规则：缺少发票、金额超标准、出差事由不清时，必须 need_human_approval=true。
输入：员工申请报销 1860 元酒店费用，标准为每晚 600 元，住了 2 晚；仅上传了支付截图，没有发票。
输出：

{"risk\_level": "high","risk\_tags": ["超标准","发票异常","材料缺失"],"need\_human\_approval": true,"reason": "住宿费1860元超标准660元（标准1200元），且未提供发票，仅支付截图，材料缺失。","missing\_materials": ["发票"]}

改进的Prompt：

角色：你是一名报销初审助手。仅根据输入材料做初步风险标记，不能批准报销。
请严格输出 JSON：
{
"risk_level": "low | medium | high",
"risk_tags": ["重复报销 | 超标准 | 发票异常 | 材料缺失 | 无明显风险"],
"need_human_approval": true,
"reason": "不超过80字",
"missing_materials": []
}

请只输出可解析的 JSON，不要使用 Markdown。

字段约束：

- risk_level 只能是 low、medium、high。
- risk_tags 只能从：重复报销、超标准、发票异常、材料缺失、无明显风险 中选择。
- need_human_approval 必须是 true 或 false，不要固定为 true。
- reason 不超过 80 个字，只能依据输入事实。
- missing_materials 列出缺失材料。

判断规则：

- 未提供发票：标记“材料缺失”，不标记“发票异常”。
- 金额超过明确报销标准：标记“超标准”。
- 同时出现两项及以上风险，risk_level 为 high。
- 仅一项一般风险，risk_level 为 medium。
- 无风险时，risk_level 为 low 且 risk_tags 为 ["无明显风险"]。
- 缺少发票、金额超标准或出差事由不清时，need_human_approval 必须为 true。

改进后的输出：

{"risk\_level":"high","risk\_tags":["超标准","材料缺失"],"need\_human\_approval":true,"reason":"费用超标准660元（标准1200元），且未提供发票，仅支付截图。","missing\_materials":["发票"]}

我学到了什么？
1、当使用json固定是输出特定格式的时候，可以分别对部分字段的取值进行限定，同时给出字段的取值范围和标准，这样能够控制输出
