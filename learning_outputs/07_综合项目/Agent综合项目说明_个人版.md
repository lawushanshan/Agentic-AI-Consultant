# Agent 综合项目说明：个人版

填写日期：2026-06-12

学习者：ASP.NET Core 程序员转 AI Solution Consultant

项目名称：客服工单分类与回复建议 Agent

---

## 1. 业务背景

企业客服团队每天需要处理大量客户问题，包括付款异常、物流咨询、退款申请、账号问题和投诉。人工处理时，客服需要阅读工单、判断类别、查询知识库、撰写回复并决定是否转人工或升级。这个流程高频、重复，且容易因人员经验不同导致处理标准不一致。

本项目目标是设计一个客服工单 Agent，帮助客服完成初步分类、优先级判断、知识库检索和回复草稿生成，同时对高风险问题进行人工确认。

---

## 2. 原流程痛点

| 痛点 | 当前影响 |
|---|---|
| 人工阅读和分类耗时 | 高峰期工单积压 |
| 分类标准不一致 | 不同客服处理结果不同 |
| 知识库查询效率低 | 回复慢，容易遗漏资料 |
| 高风险工单可能漏判 | 投诉、付款、退款问题处理风险高 |
| 回复质量不稳定 | 客户体验不一致 |

---

## 3. 目标用户

- 一线客服
- 二线客服
- 客服主管
- 客服运营人员

---

## 4. Agent 解决方案

Agent 要完成：

- 读取客户工单内容
- 判断工单类别
- 判断优先级
- 判断是否需要人工确认
- 检索知识库
- 生成回复草稿
- 输出结构化 JSON
- 记录工具调用和风险原因

Agent 不做：

- 不自动发送正式回复
- 不自动退款
- 不自动修改客户资料
- 不输出完整敏感信息
- 不绕过权限查询数据

---

## 5. 输入和输出

### 输入

```json
{
  "ticket_id": "T1001",
  "customer_id": "C001",
  "ticket_text": "我已经付款了，但是订单还是显示未支付。",
  "channel": "web"
}
```

### 输出

```json
{
  "category": "payment_issue",
  "priority": "high",
  "need_human": true,
  "suggested_reply": "您好，我们已收到您的付款异常反馈，将为您核查订单状态。",
  "risk_reason": "涉及付款异常，必须人工核查",
  "tools_used": ["get_customer_profile", "search_support_knowledge_base"],
  "references": ["客服常见问题#付款异常处理"]
}
```

---

## 6. 工作流

```text
用户提交工单
  -> Agent 理解工单内容
  -> Agent 判断类别和优先级
  -> Agent 查询客户信息和知识库
  -> Agent 生成回复建议
  -> 如果是高风险问题，要求人工确认
  -> 客服确认或修改后回复客户
  -> 系统记录分类、工具调用、审批和最终结果
```

---

## 7. Prompt 设计

```text
你是一名客服工单处理助手。
背景是：客服团队需要对客户问题进行分类、判断优先级、生成回复草稿，并识别是否需要人工确认。

你的任务：
1. 判断工单类别。
2. 判断优先级。
3. 判断是否需要人工处理。
4. 生成回复草稿。
5. 给出风险原因。

输入：
{ticket_text}

请严格按照 JSON 输出：
{
  "category": "payment_issue | logistics_issue | refund_issue | account_issue | complaint | other",
  "priority": "low | medium | high",
  "need_human": true,
  "suggested_reply": "回复草稿",
  "risk_reason": "风险原因"
}

限制条件：
- 涉及付款、退款、投诉的问题必须 need_human = true。
- 回复只能是草稿，不能声称已经完成处理。
- 不要编造订单状态、退款结果或客户资料。
```

## 10. 风险控制

- 付款、退款、投诉类工单必须人工确认。
- 回复只能生成草稿，不能自动发送。
- 客户手机号、付款流水等敏感数据必须脱敏。
- 所有工具调用都要记录日志。
- 外部文本不能覆盖系统指令。

## 12. 面试讲解稿

用 5 分钟讲清楚这个项目：

```text
我选择客服工单分类作为 Agentic AI 项目练习，因为它是一个高频、规则相对清晰、风险可控的企业场景。

原流程中，客服需要人工阅读工单、判断类别、查询知识库、写回复并决定是否升级。这个过程耗时，而且不同客服判断标准不一致。

我设计的 Agent 会先读取工单内容，判断类别和优先级，再根据需要查询客户信息和客服知识库，最后生成回复草稿。如果工单涉及付款、退款或投诉，Agent 不会自动处理，而是要求人工确认。

这个项目体现了结构化输出、工具调用、RAG 检索、人工审批、日志记录和风险控制。它不是一个单纯聊天机器人，而是一个可以嵌入客服流程的 AI 工作流原型。
```
