# AI Solution Consultant 商业化路线图

版本日期：2026-06-17

本路线图用于回答一个问题：

> 当个人完成 AI Solution Consultant 转型训练，并具备真实项目交付能力后，如何把能力产品化，形成可重复销售的咨询或交付服务。

这不是当前第 1 周要立刻执行的赚钱计划，而是 24 周学习路线之后的商业化方向。当前阶段仍然以能力、项目、作品集和交付包为优先。

---

## 1. 可复刻的商业模式本质

参考 Anthony Pierri / FletchPMM 的模式，真正值得学习的不是具体做“B2B 软件定位”，而是下面这套结构：

```text
明确细分客户
  -> 找到一个高频痛点
  -> 设计固定周期服务包
  -> 固定价格、固定交付物
  -> 用内容获客
  -> 用标准流程和内部工具提高交付效率
```

迁移到 AI Solution Consultant 方向后，对应关系是：

| FletchPMM 模式 | AI Solution Consultant 版本 |
|---|---|
| B2B 软件公司定位不清 | 企业想做 AI，但不知道做什么、怎么做、怎么控风险 |
| 两周定位冲刺 | 两周 AI Agent PoC 方案冲刺 |
| 定位文档和首页文案 | 场景清单、Agent PRD、架构图、风险清单、测试集 |
| LinkedIn 内容获客 | 技术文章、项目拆解、案例复盘、视频号/公众号/LinkedIn 内容 |
| 内部流程工具 | 场景评分器、PRD 模板、测试集生成器、风险清单生成器 |

---

## 2. 推荐商业定位

未来不建议定位成泛泛的“AI 顾问”。

推荐定位：

> 技术型 AI Agent PoC 顾问 / AI Solution Consultant，帮助企业从模糊 AI 想法变成可评审、可开发、可试点的 PoC 方案。

更具体的市场表达：

> 我帮助中小企业和 B2B 软件团队，在 2 周内判断哪些 AI Agent 场景值得做，并输出可交付的 PRD、流程、架构、风险控制和测试方案。

这个定位的优势：

- 比“卖课”更专业。
- 比“直接开发完整系统”风险更可控。
- 比“只做 Prompt 培训”壁垒更高。
- 能发挥 ASP.NET Core / 后端开发背景。
- 能直接承接当前 24 周学习路线中的所有产出。

## 3. 目标客户选择

早期不要服务所有企业，先选择容易理解、痛点明确、数据结构相对清楚的客户。

### 3.1 优先客户

| 客户类型 | 典型痛点 | 适合服务 |
|---|---|---|
| 中小 B2B 软件公司 | 想给产品加 AI，但场景模糊 | AI 功能场景诊断、Agent PRD |
| 有客服团队的企业 | 工单多、回复慢、知识库利用率低 | 客服 Agent PoC 方案 |
| 有内部知识库的公司 | 员工找制度、流程、文档困难 | RAG 知识库问答方案 |
| 销售团队 | 客户跟进、续约分析、CRM 数据利用不足 | 销售分析 Agent 方案 |
| 传统信息化企业 | 有系统、有流程，但不知道 AI 如何接入 | 流程诊断、工具调用方案 |

### 3.2 暂不优先客户

- 完全没有数字化系统的企业。
- 只想要“一个万能 AI 助手”的客户。
- 一上来要求替代核心决策的高风险场景。
- 不愿提供流程、样例数据、业务人员访谈的客户。
- 只想低价买一个 Demo，但不关心业务落地的客户。

## 4. 第一个产品化服务

建议第一个商业化服务设计为：

> AI Agent PoC 方案冲刺

### 4.1 服务对象

适合已经有业务流程、知识库、客服/销售/运营数据，但不知道 AI 场景如何落地的企业或团队。

### 4.2 周期

固定 2 周。

### 4.3 交付物

两周结束后交付：

1. 业务场景访谈纪要
2. 5 个候选 AI 场景机会清单
3. 1 个优先 PoC 场景选择说明
4. As-Is / To-Be 流程图
5. Agent PRD
6. RAG / 工具调用方案
7. 权限、审批和风险控制清单
8. 20 条测试用例
9. PoC 架构图
10. 上线前验证路线图

### 4.4 不包含内容

为了控制边界，第一个服务包不承诺：

- 不承诺完整生产系统开发。
- 不承诺上线效果指标。
- 不接管客户内部数据治理。
- 不替客户直接操作核心业务系统。
- 不替客户采购模型、云资源或第三方软件。

可以在后续服务中扩展原型开发和上线陪跑。

## 5. 服务包设计

### 5.1 入门包：AI 场景诊断 Sprint

| 项目 | 内容 |
|---|---|
| 周期 | 1 周 |
| 目标 | 判断哪些 AI 场景值得做 |
| 交付物 | 场景机会清单、评分表、优先级建议 |
| 适合客户 | 还没有明确 PoC 方向的企业 |

### 5.2 标准包：AI Agent PoC 设计 Sprint

| 项目 | 内容 |
|---|---|
| 周期 | 2 周 |
| 目标 | 输出一个可评审、可开发的 PoC 方案 |
| 交付物 | PRD、流程图、架构图、RAG/工具方案、风险清单、测试集 |
| 适合客户 | 已有候选场景，准备做 PoC 的企业 |

### 5.3 进阶包：AI Agent 原型 Sprint

| 项目 | 内容 |
|---|---|
| 周期 | 4 周 |
| 目标 | 做出可演示的最小 Agent Demo |
| 交付物 | 可运行 Demo、模拟数据、测试报告、下一步上线建议 |
| 适合客户 | 已确认 PoC 场景，愿意投入技术验证的企业 |

## 6. 早期定价策略

早期不要直接对标海外 1-3 万美元定价。先用案例验证服务质量和客户愿付费意愿。

建议分阶段：

| 阶段 | 状态 | 建议价格 |
|---|---|---|
| 练习期 | 没有真实客户案例 | 免费 / 低价试做 |
| 验证期 | 有 1-2 个模拟或熟人案例 | 3000-10000 元 / 次 |
| 成长期 | 有 3-5 个可展示案例 | 1-3 万元 / 2 周 |
| 成熟期 | 有稳定获客和复购推荐 | 按客户规模分档 |

成熟后可以考虑分档：

| 客户规模 | 方案冲刺价格 |
|---|---|
| 小团队 / 初创公司 | 1-2 万元 |
| 中小企业 | 3-5 万元 |
| 较成熟企业 | 5-10 万元 |

价格能否成立，取决于案例质量、客户痛点强度、交付可信度和获客渠道，而不是文档写得多漂亮。

## 7. 与 24 周学习路线的关系

商业化不是另起炉灶，而是从学习产出自然升级。

| 学习阶段 | 学习产出 | 未来商业化用途 |
|---|---|---|
| 第 1-4 周 | Prompt、结构化输出、API Demo | 证明能把 AI 任务变成系统输入输出 |
| 第 5-8 周 | RAG 知识库问答 Agent | 形成企业知识库方案样板 |
| 第 9-12 周 | 工具调用 Agent | 形成客服/销售流程 Agent 样板 |
| 第 13-18 周 | PRD、架构、测试、治理、上线计划 | 形成标准 PoC 方案交付能力 |
| 第 19-24 周 | 作品集、专业交付包、讲解稿 | 形成商业获客和销售材料 |

24 周结束时，至少应具备：

- 2 个可展示项目
- 1 个专业交付包
- 1 套方案讲解材料
- 1 套可复用模板
- 若干失败案例和风险处理样例

这些材料就是未来服务包的雏形。

## 8. 内容获客路线

Anthony 的关键经验是“先做分发，后做产品”。对个人转型者来说，可以更保守地执行：

```text
先记录学习过程
  -> 再输出项目拆解
  -> 再分享方法论
  -> 最后承接咨询或 PoC 服务
```

### 8.1 内容主题

建议围绕这些主题长期输出：

- ASP.NET Core 程序员如何转 AI Solution Consultant
- 企业知识库 RAG 项目复盘
- 客服工单 Agent 如何设计
- Agent 工具调用为什么不能只是暴露 API
- AI 项目为什么必须有测试集
- Prompt Injection 和企业权限风险
- 一个 Agent PRD 应该怎么写
- AI PoC 到上线之间差什么

### 8.2 内容形式

| 渠道 | 内容形式 |
|---|---|
| 公众号 / 博客 | 长文复盘、教程、项目拆解 |
| 视频号 / B 站 | 项目演示、学习周记、方案讲解 |
| GitHub / Gitee | Demo、模板、交付包样例 |
| LinkedIn / 即刻 / 知识星球 | 短观点、案例片段、方法论 |
| Newsletter | 月度学习和项目复盘 |

### 8.3 早期内容节奏

第 1-12 周：

- 每周 1 篇学习复盘
- 每两周 1 篇项目技术笔记

第 13-24 周：

- 每周 1 篇交付方法论
- 每月 1 篇完整项目复盘

24 周后：

- 每周 2-3 条短内容
- 每月 1 篇深度案例
- 每季度更新一次服务包说明

## 9. 内部工具路线

未来要把服务做得可重复，必须把流程工具化。

### 9.1 第一批内部工具

| 工具 | 用途 |
|---|---|
| AI 场景评分器 | 对候选场景按业务价值、数据可得、风险可控、技术可行性打分 |
| Agent PRD 生成器 | 根据访谈信息生成 PRD 初稿 |
| RAG 方案检查器 | 检查文档、权限、引用、拒答设计是否完整 |
| 工具调用设计器 | 生成工具名称、输入输出、权限、风险等级 |
| 测试集生成器 | 生成正常、边界、权限、失败、注入攻击测试用例 |
| 风险清单生成器 | 输出审批、日志、禁止动作、敏感数据处理建议 |

### 9.2 工具建设原则

- 先服务自己的交付流程。
- 先做内部工具，不急着 SaaS 化。
- 每个工具都要对应一个固定交付物。
- 工具输出必须由人审，不直接交给客户。
- 工具越标准，服务越容易复制。

## 10. 12 个月商业化时间线

### 第 0-6 个月：能力和作品期

目标：

- 完成 24 周学习路线。
- 做出 2 个项目。
- 形成 1 个专业交付包。
- 开始记录公开内容。

不做：

- 不急着收费卖咨询。
- 不包装成专家。
- 不接高风险生产项目。

### 第 7-9 个月：小范围验证期

目标：

- 找 1-3 个熟人、朋友公司或模拟客户做诊断。
- 验证“AI Agent PoC 方案冲刺”的交付流程。
- 记录客户问题和交付反馈。
- 改进模板和内部工具。

可尝试：

- 免费诊断
- 低价试做
- 公开复盘匿名案例

### 第 10-12 个月：产品化试售期

目标：

- 固定服务包名称。
- 固定交付物。
- 固定流程。
- 形成第一版服务介绍页。
- 尝试收取标准价格。

关键指标：

- 是否有人愿意付费
- 是否能 1-2 周交付
- 是否能产生转介绍
- 是否能复用 70% 以上流程

## 11. 风险和约束

### 风险 1：过早商业化

表现：

- 没有项目案例就开始卖咨询。
- 只能讲概念，无法交付方案。
- 客户一追问技术细节就露怯。

控制方式：

- 24 周内以学习和案例为主。
- 第一个收费服务前至少完成 2 个项目和 1 个交付包。

### 风险 2：服务范围过大

表现：

- 客户让做完整系统、上线、运维、培训、数据治理，全部接。

控制方式：

- 第一阶段只卖诊断和 PoC 方案。
- 原型开发和上线陪跑作为后续服务。

### 风险 3：只依赖个人信任

表现：

- 所有线索都来自个人关系。
- 服务无法由其他人执行。

控制方式：

- 从第一天就沉淀模板、流程、工具和评审标准。
- 每次交付都保留可复用资产。

### 风险 4：内容输出偏空

表现：

- 只写趋势、观点、鸡汤。
- 没有项目细节、失败案例和交付物。

控制方式：

- 内容必须来自真实练习、项目和复盘。
- 每篇内容尽量包含输入、过程、输出、问题和改进。

## 12. 当前阶段行动建议

现在不要立刻卖服务。

当前最重要的是：

1. 按 24 周计划完成学习。
2. 每周补齐 `learning_outputs` 个人版产出。
3. 关键节点调用 AI 学习导师评审。
4. 从第 1 周开始记录公开学习笔记。
5. 把未来商业化服务需要的模板同步沉淀下来。

当前可做的最低成本商业化准备：

- 给每个学习产出都写清业务背景。
- 记录所有失败案例。
- 把 Prompt、流程图、PRD、测试集做成可复用模板。
- 学会用 5 分钟讲清一个项目。
- 每月写一篇完整复盘。

---

## 13. 最终目标

商业化的最终目标不是“靠 AI 热点赚钱”，而是形成一套可重复的专业服务：

> 帮助企业把模糊 AI 想法，转化为可评审、可开发、可测试、可控风险的 Agent PoC 方案。

当你能稳定完成这件事，再去考虑课程、训练营、咨询、企业 PoC、工具产品或团队化交付。
