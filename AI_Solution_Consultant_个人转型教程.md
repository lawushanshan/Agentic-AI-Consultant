# AI Solution Consultant 个人转型教程

版本日期：2026-06-12

适合对象：

- 想从纯开发转向 AI 解决方案、AI 顾问、Agentic AI 交付方向
- 希望保留技术优势，同时补齐业务、方案、交付、沟通能力

本教程目标：

> 帮助你从“会写代码的程序员”逐步升级为“能做 AI PoC、能设计 Agent 方案、能对接业务和系统、能推动上线的 AI Solution Consultant”。

这不是一份只用来阅读的文档。你应该按阶段执行，每个阶段都要产出可检查的作品。最终目标是形成一套可以求职、转岗、面试、接项目使用的个人作品集和交付包。

---

## 1. 你的职业定位

### 1.1 推荐定位

你的最佳职业定位不是泛泛的 AI 顾问，而是：

> 技术型 AI Solution Consultant / Agentic AI Delivery Consultant

更具体地说：

> 你要成为一个既能写代码做原型，又能理解业务流程、设计 Agent 方案、接入企业系统、控制权限风险、评估效果和推动上线的人。

### 1.2 不建议的定位

不要把自己定位成：

- 只会写 Prompt 的 Prompt 工程师
- 只会讲概念的 AI 培训师
- 只做 PPT 的咨询顾问
- 只关注模型算法的研究型工程师
- 只做普通 CRUD 的业务程序员

这些方向要么门槛会被工具拉低，要么与你当前程序员背景的优势不匹配。

### 1.3 推荐岗位名称

你后续可以关注这些岗位：

- AI Solution Consultant
- AI Solutions Engineer
- AI Solution Architect
- LLM Application Engineer
- Agentic AI Consultant
- AI Implementation Consultant
- Forward Deployed AI Engineer
- 大模型应用解决方案工程师
- 企业 AI 应用交付顾问
- AI 产品技术顾问

### 1.4 你的核心卖点

你未来应该向市场表达这句话：

> 我不是只会调用模型，而是能把 LLM、RAG、Agent、工具调用、权限、日志、审批、评估接入企业真实流程。

---

## 2. 总体成长路线

建议按 24 个月规划，但前 6 个月必须看到作品。

```text
普通程序员
  -> LLM 应用开发者
  -> AI Solution Engineer
  -> AI Solution Consultant
  -> Agentic AI Solution Architect / FDE
```

阶段总览：

| 阶段 | 时间 | 目标 | 核心成果 |
|---|---|---|---|
| 阶段 0 | 第 0 周 | 建立职业地图 | 个人能力盘点和转型计划 |
| 阶段 1 | 第 1-4 周 | 掌握 LLM 应用基础 | Prompt、结构化输出、API 小项目 |
| 阶段 2 | 第 5-8 周 | 完成 RAG 入门项目 | 企业知识库问答 Agent |
| 阶段 3 | 第 9-12 周 | 完成工具调用 Agent | 客服工单或销售分析 Agent |
| 阶段 4 | 第 13-18 周 | 补齐方案和交付能力 | Agent PRD、流程图、风险清单、测试集 |
| 阶段 5 | 第 19-24 周 | 形成第一个完整交付包 | 可展示的 AI Solution Consultant 作品集 |
| 阶段 6 | 第 7-12 个月 | 行业化和求职准备 | 2-3 个项目作品、面试话术 |
| 阶段 7 | 第 13-24 个月 | 真实交付能力升级 | 真实项目经验、上线和运营能力 |

---

## 3. 每周固定学习节奏

建议每周投入 8-10 小时。

| 时间 | 内容 | 要求 |
|---|---|---|
| 2 小时 | 学 AI 技术 | LLM、RAG、Agent、工具调用、评估 |
| 2 小时 | 写代码做项目 | 每周必须推进一个可运行功能 |
| 2 小时 | 学业务和交付 | 流程图、PRD、ROI、风险治理 |
| 1 小时 | 测试和复盘 | 记录失败案例，不只记录成功 |
| 1 小时 | 整理作品集 | 把代码、文档、截图、评估结果沉淀下来 |

学习原则：

1. 每周必须有产出，不能只看教程。
2. 每个项目必须有业务背景，不能只有代码。
3. 每个 Agent 项目都要有测试集和风险说明。
4. 每次失败都要记录原因和修复方式。
5. 每个月至少整理一次作品集。

## 5. 阶段 1：LLM 应用基础

时间：第 1-4 周

### 5.1 阶段目标

你要从“会使用 AI 工具”升级为“能用代码调用模型、控制输出格式、构建最小 AI 应用”的开发者。

### 5.2 需要掌握的知识

- Prompt 基础
- 结构化输出
- JSON Schema
- LLM API 调用
- Token、成本、延迟
- 模型输出不稳定的处理方式
- 基础错误处理和日志

### 5.3 第 1 周：Prompt 和任务拆解

任务：

1. 写 10 个业务型 Prompt。
2. 每个 Prompt 都要包含角色、背景、任务、输入、输出格式、限制条件。
3. 选择 3 个效果不好的 Prompt，记录为什么失败。
4. 改写失败 Prompt，再记录改进结果。

推荐练习场景：

- 客服问题分类
- 报销材料检查
- 销售邮件生成
- 会议纪要转行动项
- 合同风险摘要

对应模板：

```text
learning_outputs/02_Prompt练习/Prompt练习集_模板.md
```

阶段小成果：

- 至少 10 个 Prompt 案例
- 至少 3 个改进前后对比

### 5.4 第 2 周：结构化输出

任务：

1. 让模型输出固定 JSON。
2. 设计 5 个业务字段结构。
3. 测试模型是否稳定输出字段。
4. 记录字段缺失、格式错误、内容不一致的案例。

示例输出：

```json
{
  "category": "payment_issue",
  "priority": "high",
  "need_human": true,
  "reason": "客户已付款但订单仍显示未支付"
}
```

你要理解：

- 为什么企业系统需要结构化输出
- 哪些字段必须枚举
- 哪些字段需要置信度
- 输出格式错误时怎么重试或修复

阶段小成果：

- 5 个结构化输出案例
- 1 份字段设计说明

### 5.5 第 3 周：模型 API 调用

任务：

1. 用你熟悉的语言写一个命令行 AI 小助手。
2. 输入一段业务文本。
3. 调用模型 API。
4. 输出结构化 JSON。
5. 把请求、响应、错误记录到日志。

最小功能：

```text
用户输入客服问题
  -> 调用模型
  -> 输出类别、优先级、是否转人工、理由
  -> 保存日志
```

阶段小成果：

- 一个可运行命令行程序
- 一份 README
- 至少 10 条测试输入

### 5.6 第 4 周：小型 AI 应用封装

任务：

1. 把第 3 周命令行程序封装成简单 Web API。
2. 提供一个 `/classify-ticket` 接口。
3. 输入 JSON，输出 JSON。
4. 增加异常处理。
5. 增加请求日志。

接口示例：

```http
POST /classify-ticket
Content-Type: application/json
```

```json
{
  "ticket_text": "我已经付款了，但订单还是未支付"
}
```

返回：

```json
{
  "category": "payment_issue",
  "priority": "high",
  "need_human": true,
  "reason": "涉及付款异常，需要人工核查"
}
```

### 5.7 阶段成果

- Prompt 练习集
- 结构化输出案例
- 一个模型 API 调用小程序
- 一个简单 AI Web API

### 5.8 验收标准

合格标准：

- 能解释 Prompt 为什么要包含背景、任务、格式和限制
- 能让模型输出稳定 JSON
- 能用代码调用模型 API
- 能记录输入、输出和错误日志
- 能说明模型输出为什么不能直接进入高风险业务系统

## 6. 阶段 2：RAG 企业知识库问答 Agent

时间：第 5-8 周

### 6.1 阶段目标

完成第一个可展示 AI 项目：企业知识库问答 Agent。

这个项目用于证明你理解：

- 文档问答为什么不能只靠模型常识
- RAG 如何检索资料
- 回答为什么要带引用
- 找不到答案时为什么要拒答
- 文档权限和版本为什么重要

### 6.2 项目场景

推荐选择下面一个场景：

- 员工制度问答
- 产品手册问答
- 客服 FAQ 问答
- 技术文档助手
- 公司内部流程问答

如果没有真实资料，可以自己准备 5-10 篇 Markdown 文档模拟企业知识库。

### 6.3 第 5 周：准备文档和问题集

任务：

1. 准备 5-10 篇 Markdown 或 PDF 文档。
2. 为每篇文档记录标题、版本、负责人、权限等级。
3. 写 20 个用户问题。
4. 为每个问题写期望答案来源。

对应模板：

```text
learning_outputs/04_RAG原型/RAG原型说明_模板.md
```

阶段小成果：

- 文档清单
- 20 个测试问题
- 期望答案来源表

### 6.4 第 6 周：实现最小 RAG 流程

任务：

1. 读取文档。
2. 文档切分。
3. 生成 embedding。
4. 存入向量数据库或本地索引。
5. 用户提问时检索相关片段。
6. 把片段交给模型回答。

最小流程：

```text
用户问题
  -> 检索相关文档片段
  -> 组装上下文
  -> 模型基于上下文回答
  -> 返回答案和引用来源
```

阶段小成果：

- 可运行 RAG Demo
- 至少支持 5 篇文档
- 回答中带引用来源

### 6.5 第 7 周：加入拒答和引用校验

任务：

1. 如果检索不到相关文档，要求 Agent 回答“不知道”。
2. 回答必须引用文档标题或 chunk_id。
3. 测试 5 个文档中没有答案的问题。
4. 记录模型是否编造答案。

你要重点观察：

- 检索结果是否相关
- 模型是否基于资料回答
- 引用是否真的支持答案
- 无答案时是否拒答

阶段小成果：

- 5 个无答案测试
- 5 个错误案例复盘
- 拒答策略说明

### 6.6 第 8 周：整理成项目说明

任务：

1. 写项目 README。
2. 画系统流程图。
3. 整理测试结果。
4. 写权限和风险说明。
5. 准备 3 分钟演示话术。

项目说明结构：

```markdown
# 企业知识库问答 Agent

## 业务背景
## 原流程痛点
## RAG 解决方案
## 系统流程
## 文档清单
## 测试问题
## 测试结果
## 错误案例
## 权限和风险
## 下一步优化
```

### 6.7 阶段成果

- 企业知识库问答 Agent Demo
- RAG 原型说明
- 20 个测试问题
- 错误案例复盘
- 项目 README

### 6.8 验收标准

合格标准：

- 能解释 RAG 的基本流程
- 回答能带引用
- 找不到答案时不编造
- 至少记录 5 个错误案例
- 能说明文档权限、版本和负责人为什么重要

## 7. 阶段 3：工具调用型 Agent

## 8. 阶段 4：方案和交付能力

时间：第 13-18 周

### 8.1 阶段目标

从“能做 Demo”升级为“能写方案、能交付 PoC、能和业务方沟通”。

你要补齐的是 AI Solution Consultant 的核心交付能力。

### 8.2 第 13 周：场景机会识别

任务：

1. 列出 5 个候选 AI 场景。
2. 从业务价值、频率、数据可得、风险可控、技术可行性打分。
3. 选出 1 个 PoC 优先场景。

评分维度：

| 维度 | 说明 |
|---|---|
| 业务价值 | 是否节省时间、降低成本、提升收入或降低风险 |
| 频率规模 | 任务是否高频，影响人数是否足够 |
| 任务清晰度 | 输入、输出、判断规则是否明确 |
| 数据可得性 | 数据是否能合法访问 |
| 风险可控性 | 错误是否能发现、纠正、回滚 |
| 技术可行性 | 现有技术是否能较快实现 |

阶段小成果：

- 场景机会清单
- PoC 场景选择说明

### 8.3 第 14 周：Agent PRD

任务：

为你选择的项目写一份 Agent PRD。

## 12. 学习过程中最容易犯的错误

### 错误 1：只学技术，不学业务

表现：

- 只会说 RAG、Agent、embedding
- 讲不清项目解决什么业务问题
- 不知道用户为什么要用

修正：

- 每个项目都写业务背景、原流程痛点、成功指标和 ROI

### 错误 2：只做 Demo，不做评估

表现：

- 演示时看起来不错
- 没有测试集
- 无法证明准确率和稳定性

修正：

- 每个项目至少 20 条测试用例
- 记录失败案例
- 每次改 Prompt 或模型后做回归测试

### 错误 3：忽略权限和审批

表现：

- Agent 能随便查数据
- Agent 能直接发邮件或改系统
- 没有日志和审计

修正：

- 所有工具都要设计风险等级
- 高风险动作必须人工确认
- 工具调用必须记录日志

### 错误 4：作品集只有代码

表现：

- GitHub 有代码，但没有业务文档
- 面试官看不出你的顾问能力

修正：

- 每个项目都补 PRD、流程图、架构图、测试集、风险清单和项目讲解稿

### 错误 5：想一步成为顾问

表现：

- 还没有项目，就开始包装概念
- 缺少可运行 Demo

修正：

- 先做 LLM 应用开发者
- 再做 AI Solution Engineer
- 最后升级为 AI Solution Consultant
