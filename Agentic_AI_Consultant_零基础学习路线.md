# Agentic AI Consultant 零基础学习路线

版本日期：2026-06-11

适合人群：

- 非计算机专业刚毕业的大学生
- 没有系统编程经验，但愿意学习 AI 落地的人
- 做过运营、市场、行政、客服、销售、财务、人事、项目助理等工作，想转向 AI 顾问方向的人
- 看过《Agentic AI Consultant 就职与学习手册》，但不知道下一步怎么学的人

学习目标：

> 用 12 周时间，从“只听过 AI”成长到“能做出 1 个可展示的 Agent 项目，并能讲清业务价值、技术逻辑和风险控制”。

本路线不要求你一开始会复杂编程。你会先从职业理解、Prompt、业务流程开始，再逐步进入 API、RAG、工具调用和项目作品集。

---

## 1. 先说清楚：零基础学习难在哪里

对新人来说，Agentic AI Consultant 的难点不是某一个知识点特别难，而是知识跨度很大。

你会同时遇到：

- AI 概念
- 业务流程
- Prompt
- API
- 数据
- 知识库
- Agent
- 安全风险
- 项目交付
- 面试表达

如果没有顺序，很容易学乱。

所以这份路线的原则是：

1. 先理解，再动手
2. 先小任务，再完整项目
3. 先能讲清楚，再追求复杂技术
4. 每周必须有产出
5. 产出最后都能放进作品集

---

## 2. 你最终要达到什么水平

12 周后，你应该能做到：

- 用自己的话解释 Agent、RAG、工具调用、人工审批
- 写出清晰的 Prompt，让 AI 按指定格式完成任务
- 看懂一个简单业务流程，并找出适合 AI 辅助的环节
- 做一个简单知识库问答 Demo
- 做一个简单工具调用 Demo
- 设计一个带风险控制的业务 Agent 原型
- 写一份项目说明文档
- 在面试中讲清楚自己的项目

你暂时不需要做到：

- 独立开发企业级 Agent 平台
- 熟练掌握所有云服务
- 深入训练模型
- 精通复杂多 Agent 框架
- 独立负责大型客户上线

新人阶段的目标是完成从 0 到 1，不是一步成为专家。

## 5. 第 1 周：理解这个职业

### 5.1 学习目标

你要能回答：

- Agentic AI Consultant 是什么
- 它和普通 AI 使用者有什么区别
- 它和产品经理、售前、工程师、FDE 有什么关系
- 为什么企业需要这个角色

### 5.2 学习内容

重点阅读总纲手册中的：

- 第 1 章：这个职业到底是什么
- 第 2 章：Agentic AI Consultant 和 FDE 的关系
- 第 3 章：你每天可能在做什么

### 5.3 小白解释

你可以这样理解：

> 这个职业不是只负责让 AI 回答问题，而是负责把 AI 放进企业真实工作流程里，让它帮人完成任务，同时保证安全、可控、有价值。

### 5.4 本周练习

找 3 个你熟悉的工作场景，例如：

- 学校社团报名
- 公司报销
- 客服回复
- 招聘筛简历
- 销售跟进客户

对每个场景回答：

1. 谁在做？
2. 每天或每周做几次？
3. 哪些步骤重复？
4. 哪些步骤需要判断？
5. AI 可以帮什么？
6. 哪些地方必须人来确认？

### 5.5 本周产出

写一份《Agentic AI Consultant 职业理解一页纸》：

```markdown
# Agentic AI Consultant 职业理解

## 这个岗位解决什么问题

## 每天主要做什么

## 需要哪些能力

## 我和这个岗位的差距

## 接下来 12 周我要补什么
```

## 6. 第 2 周：理解大模型能力边界

### 6.1 学习目标

你要知道：

- 大模型擅长什么
- 大模型不擅长什么
- 为什么模型会编造
- 为什么不能把 AI 输出直接当事实

### 6.2 大模型擅长

大模型擅长：

- 总结文本
- 改写内容
- 分类
- 提取信息
- 生成草稿
- 解释概念
- 根据例子模仿格式
- 帮你拆解任务

### 6.3 大模型不擅长

大模型不擅长：

- 保证事实永远正确
- 自动知道企业内部最新数据
- 自己判断权限
- 对高风险业务结果负责
- 替代专业审批
- 在没有资料时可靠回答

### 6.4 本周练习

让 AI 完成 5 类任务：

1. 总结一篇文章
2. 把一段话改成正式邮件
3. 从文本中提取姓名、日期、金额
4. 判断 10 条客服问题属于什么类别
5. 让 AI 回答一个它可能不知道的内部问题

记录：

- 哪些回答很好
- 哪些回答不可靠
- 哪些需要补充资料
- 哪些需要人确认

### 6.5 本周产出

写一份《AI 能力边界笔记》：

```markdown
# AI 能力边界笔记

## AI 擅长的任务

## AI 容易出错的任务

## 我观察到的 5 个错误案例

## 企业使用 AI 时必须注意什么
```

## 7. 第 3 周：Prompt 基础

### 7.1 学习目标

你要学会写清楚任务。

一个好 Prompt 通常包含：

- 角色
- 背景
- 任务
- 输入
- 输出格式
- 限制条件
- 示例

### 7.2 基础模板

```text
你是一名【角色】。
背景是：【业务背景】。
你的任务是：【具体任务】。
输入内容是：【输入】。
请按照以下格式输出：【格式】。
注意事项：【限制条件】。
```

### 7.3 示例

```text
你是一名客服主管。
背景是：我们需要把客户问题分配给不同客服小组。
你的任务是：判断下面的客户问题属于哪个类别，并给出优先级。
输入内容是：客户说“我昨天付款了，但是订单还显示未支付。”
请按照以下格式输出：
类别：
优先级：
判断理由：
是否需要转人工：
注意事项：如果涉及付款、退款、投诉，优先级至少为中。
```

### 7.4 本周练习

写 20 个 Prompt，覆盖：

- 总结
- 分类
- 提取
- 改写
- 生成邮件
- 生成表格
- 风险判断
- 流程拆解
- 面试问题生成
- 客服回复

每个 Prompt 都要记录：

- 原始 Prompt
- AI 输出
- 你哪里不满意
- 修改后的 Prompt
- 改进结果

### 7.5 本周产出

整理《20 个 Prompt 练习案例》。

每个案例格式：

```markdown
## 案例 1：客服问题分类

### 原始 Prompt

### 输出结果

### 存在问题

### 改进 Prompt

### 改进后结果
```

## 8. 第 4 周：结构化输出

### 8.1 学习目标

你要学会让 AI 输出稳定格式。

企业项目中，AI 输出经常要进入系统，所以不能只是自然语言。

常见格式：

- 表格
- 清单
- JSON
- 固定字段
- 分级结果

### 8.2 为什么结构化输出重要

如果 AI 输出：

```text
这个客户问题比较紧急，可能需要客服尽快处理。
```

人能看懂，但系统不好处理。

如果 AI 输出：

```json
{
  "category": "payment_issue",
  "priority": "high",
  "need_human": true,
  "reason": "客户已付款但订单状态异常"
}
```

系统就可以继续处理。

### 8.3 本周练习

做 5 个结构化任务：

1. 客服问题分类为 JSON
2. 简历信息提取为表格
3. 报销申请风险判断为固定字段
4. 会议纪要转行动项清单
5. 合同条款提取为风险列表

### 8.4 本周产出

整理《结构化输出练习》。

你要能说明：

- 为什么这个任务需要结构化输出
- 字段怎么设计
- 哪些字段必须有
- AI 输出不稳定时怎么改 Prompt

## 9. 第 5 周：业务流程理解

### 9.1 学习目标

你要学会看懂一个工作流程。

Agent 不是凭空存在的，它要进入业务流程。

### 9.2 流程拆解模板

每个流程都问：

- 谁触发？
- 输入是什么？
- 谁处理？
- 用什么系统？
- 中间有哪些判断？
- 输出是什么？
- 哪些地方容易出错？
- 哪些地方可以 AI 辅助？
- 哪些地方必须人工审批？

### 9.3 示例：报销流程

```text
员工提交报销单
  -> 系统检查发票
  -> 财务初审
  -> 部门负责人审批
  -> 财务付款
  -> 员工收到款项
```

AI 可以帮：

- 检查材料是否齐全
- 提取发票金额和日期
- 判断是否超过标准
- 标记异常报销

AI 不应该自动做：

- 最终批准付款
- 修改财务记录
- 删除报销材料

### 9.4 本周练习

选择一个你熟悉的流程：

- 报销
- 请假
- 客服工单
- 招聘筛选
- 采购申请
- 会议纪要跟进

画出流程图，并标注：

- AI 可辅助节点
- 人工确认节点
- 数据来源
- 风险点

### 9.5 本周产出

一张《业务流程图》和一份说明。

## 10. 第 6 周：场景选择和 ROI

### 10.1 学习目标

你要学会判断一个场景值不值得做。

不是所有任务都适合 Agent。

### 10.2 场景评分表

每项 1-5 分：

| 维度 | 说明 |
|---|---|
| 频率 | 任务发生是否频繁 |
| 人工耗时 | 是否明显浪费时间 |
| 规则清晰 | 专家是否能讲清判断标准 |
| 数据可得 | 所需数据是否能拿到 |
| 风险可控 | 出错后是否可发现、可纠正 |
| 负责人明确 | 是否有人愿意推动 |

总分越高，越适合先做。

### 10.3 简单 ROI 算法

```text
每月节省小时 = 每月任务次数 * 单次节省分钟 / 60
每月节省金额 = 每月节省小时 * 人均小时成本
```

例子：

```text
每月 1000 个客服工单
每个工单 AI 节省 3 分钟
每月节省小时 = 1000 * 3 / 60 = 50 小时
如果人均小时成本 80 元
每月节省金额 = 50 * 80 = 4000 元
```

### 10.4 本周练习

列出 5 个你观察到的 AI 场景，用评分表打分。

### 10.5 本周产出

《Agent 场景评分表》。

## 11. 第 7 周：RAG 入门

### 11.1 学习目标

你要理解“知识库问答”是怎么工作的。

RAG 的核心是：

> 先找资料，再回答。

### 11.2 不写代码的练习方式

你可以先用支持知识库上传的 AI 工具练习。

准备：

- 3-5 篇制度、产品说明或 FAQ 文档
- 10 个用户问题
- 期望答案

练习：

1. 上传文档
2. 提问
3. 检查回答是否引用文档
4. 记录错误
5. 修改问题或文档结构

### 11.3 要观察什么

重点观察：

- AI 是否真的根据文档回答
- 回答有没有编造
- 找不到答案时是否承认不知道
- 引用是否准确
- 文档写得乱会不会影响答案

### 11.4 本周产出

《制度问答 Agent 原型说明》：

```markdown
# 制度问答 Agent 原型

## 使用文档

## 目标用户

## 典型问题

## 测试结果

## 错误案例

## 改进方向
```

## 12. 第 8 周：API 入门

### 12.1 学习目标

你不一定马上会写复杂代码，但要理解 API 是什么。

API 可以理解为：

> 系统和系统之间说话的接口。

比如：

- Agent 查询订单系统
- Agent 调用 CRM
- Agent 创建工单
- Agent 发送消息

这些都需要 API。

### 12.2 必懂概念

| 概念 | 小白解释 |
|---|---|
| HTTP | 系统通信的规则 |
| GET | 查询数据 |
| POST | 提交数据 |
| JSON | 系统交换数据的常见格式 |
| Token | 身份凭证 |
| Error Code | 出错时的编号 |

### 12.3 最小练习

不用急着写复杂代码。先读懂一个 API 示例：

```http
GET /customers/123/orders
Authorization: Bearer token_xxx
```

它的意思是：

> 查询客户 123 的订单，调用时需要带身份凭证。

返回可能是：

```json
{
  "customer_id": "123",
  "orders": [
    {
      "order_id": "A001",
      "amount": 5000,
      "status": "paid"
    }
  ]
}
```

### 12.4 本周练习

设计 3 个你希望 Agent 调用的 API：

1. 查询客户订单
2. 查询员工制度
3. 创建客服工单

每个 API 写清楚：

- 用途
- 输入
- 输出
- 谁有权限调用
- 是否有风险

### 12.5 本周产出

《API 概念练习卡片》。

## 13. 第 9 周：工具调用

### 13.1 学习目标

你要理解 Agent 如何使用工具。

Agent 本身不知道企业数据，也不能凭空操作系统。它需要工具。

工具可以是：

- 查询订单
- 查询库存
- 创建工单
- 发送消息
- 生成报表

### 13.2 工具设计卡片

每个工具都要写：

```markdown
# 工具名称

## 用途

## 输入参数

## 输出结果

## 风险等级

## 是否需要人工确认

## 失败时怎么办
```

### 13.3 示例

```markdown
# 查询客户订单工具

## 用途
用于续约风险分析时查询客户最近 12 个月订单。

## 输入参数
customer_id
start_date
end_date

## 输出结果
订单数量、订单金额、付款状态、最近订单时间。

## 风险等级
中

## 是否需要人工确认
不需要，但必须记录日志。

## 失败时怎么办
提示用户无法查询，并建议联系管理员。
```

### 13.4 本周练习

围绕“销售续约分析 Agent”设计 5 个工具：

- 查询客户资料
- 查询订单记录
- 查询投诉记录
- 生成邮件草稿
- 保存跟进记录

### 13.5 本周产出

《销售续约分析 Agent 工具卡片》。

## 14. 第 10 周：风险控制

### 14.1 学习目标

你要知道 Agent 哪些事情可以自动做，哪些必须人工确认。

### 14.2 风险分级

| 等级 | 示例 | 控制方式 |
|---|---|---|
| 低风险 | 总结文档、分类问题 | 记录日志 |
| 中风险 | 查询客户数据、生成建议 | 权限校验 |
| 高风险 | 发邮件、提交订单 | 人工确认 |
| 极高风险 | 付款、删除数据、法律承诺 | 禁止或多级审批 |

### 14.3 本周练习

选择一个项目，列出：

- Agent 可以自动做的事
- Agent 只能建议、不能执行的事
- Agent 必须请求人工确认的事
- Agent 绝对不能做的事

《Agent 风险控制清单》。

## 15. 第 11 周：综合项目

### 15.1 学习目标

完成一个小型 Agent 项目方案。

推荐项目：

- 企业制度问答 Agent
- 客服工单分类 Agent
- 销售续约分析 Agent

新人最推荐：

> 客服工单分类 Agent。

因为它数据容易模拟，流程容易理解，风险也比较好控制。

## 17. 新人最容易犯的 8 个错误

### 错误 1：只收藏资料，不动手

解决方法：

每周必须有一个文件产出。

### 错误 2：一开始就学复杂框架

解决方法：

先会 Prompt、流程、RAG、工具设计，再学框架。

### 错误 3：以为会用 AI 就能做顾问

解决方法：

顾问要能分析业务、设计方案、控制风险。

### 错误 4：项目只做功能，不写文档

解决方法：

每个项目都写背景、方案、风险、评估。

### 错误 5：不记录失败案例

解决方法：

失败案例最能体现你是否理解 Agent 的边界。

### 错误 6：忽略权限和审批

解决方法：

任何涉及客户数据、财务、邮件、订单的动作都要设计权限。

### 错误 7：只讲技术，不讲价值

解决方法：

每个项目都要回答：节省了什么、提升了什么、降低了什么风险。

### 错误 8：期待一次学完

解决方法：

把目标改成：每 2 周完成一个小闭环。

## 20. 零基础结业验收标准

完成这份路线后，不以“看完文档”为结业标准，而以“能拿出成果并讲清楚”为标准。

### 20.1 必须提交的成果

| 编号 | 成果 | 合格标准 |
|---|---|---|
| 1 | 职业理解一页纸 | 能说明岗位职责、能力要求、自己差距 |
| 2 | AI 能力边界笔记 | 至少记录 5 个 AI 做得好和做错的案例 |
| 3 | Prompt 练习集 | 至少 20 个 Prompt，并有改进前后对比 |
| 4 | 结构化输出练习 | 至少 5 个固定格式输出案例 |
| 5 | 业务流程图 | 至少 1 个流程，标出 AI 可介入点和人工审批点 |
| 6 | 场景评分表 | 至少比较 5 个候选场景 |
| 7 | RAG 原型说明 | 至少包含文档、问题、回答效果、错误案例 |
| 8 | 工具设计卡片 | 至少 5 个工具，说明输入、输出、权限、风险 |
| 9 | 风险控制清单 | 至少区分自动执行、人工确认、禁止执行三类动作 |
| 10 | 综合项目说明 | 能讲清业务背景、方案、测试结果、风险和下一步 |

### 20.2 口头讲解验收

学习者应该能用 5-8 分钟讲清楚一个项目：

```text
我选择的场景是什么？
原来的业务流程有什么问题？
Agent 介入后流程怎么变？
我设计了哪些输入和输出？
哪些动作可以自动做？
哪些动作必须人工确认？
我怎么测试它？
它还有哪些不足？
```

如果能讲清楚这些问题，说明已经具备入门级 Agentic AI Consultant 的基本思维。

### 20.3 不合格信号

如果出现以下情况，说明还需要回炉：

- 只能解释概念，拿不出练习产出
- 项目没有业务背景，只有 AI 对话截图
- 不知道 Agent 什么时候会出错
- 没有任何测试数据
- 没有权限、审批、风险意识
- 讲不清项目能节省什么或改善什么
